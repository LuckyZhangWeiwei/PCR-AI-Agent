# AI Agent 历史上下文与 Token 限制说明

## 三个关键参数

| 参数 | 默认值 | 配置位置 | 作用范围 |
|------|--------|----------|----------|
| `toolResultMaxChars` | 12000 字符 | Settings → 工具结果最大字符数 | 当轮分析 |
| `toolResultMaxHistoryChars` | 6000 字符 | Settings → 历史存储上限 | 多轮历史 |
| `SUMMARIZE_THRESHOLD` | 40 条消息 | 后端固定 | 历史压缩触发点 |

---

## 1. toolResultMaxChars（12000）—— 当轮分析上限

工具执行完毕后，返回结果**发给 LLM 进行分析时**的字符上限。

- 超过 12000 字符的部分会被截断，LLM 看不到
- JB 查询超限时自动切换紧凑格式（保留摘要字段，省略 rows 明细）
- **只影响当前这一轮**，分析完就过去了

## 2. toolResultMaxHistoryChars（6000）—— 历史存储上限

同一条工具结果**存入会话历史**时的字符上限。

- 历史里每条工具消息最多保留 6000 字符
- 后续每一轮 LLM 调用都要带上完整历史，所以历史条目越精简越好
- 6000 < 12000 是刻意的：当轮分析值得多花 token，历史里的残留要节省

> **注意**：两者截断的是**同一条**工具结果，只是截断长度不同。
> 历史里存的是残缺版（6000），LLM 当轮看到的是更完整版（12000）。

## 3. 历史压缩（触发阈值：40 条消息）

当会话历史超过 **40 条消息**时，自动触发压缩：

1. 取出最早的约 20 条消息
2. 额外调用一次 LLM，把这 20 条压缩成一段摘要文字
3. 删除原始 20 条，改用摘要替代
4. 保留最近 20 条消息原文不变

**压缩的代价：**
- 摘要会丢失细节，lot ID、BIN 编号等精确数字可能被"洗掉"
- 压缩本身需要额外调一次 LLM，有延迟

**实际触发频率：**
正常一次聊天 5~15 轮，历史消息约 10~30 条，基本不会触发。
这个机制是极端长对话（几十轮）的兜底保障。

---

## 每轮发给 LLM 的完整内容

### 压缩触发前（历史 ≤ 40 条，正常情况）

```
┌─────────────────────────────────────────────────┐
│ system prompt（固定，~2000 token）               │
│ ─────────── 全部历史消息（完整原文）────────────  │
│ user: "WA03P02G 最近触发情况"                    │
│ assistant: "我来查一下..."                       │
│ tool result: [最多 6000 字符]  ← historyChars   │
│ assistant: "结论：共触发 12 次..."               │
│ user: "BIN7 具体哪几片"                          │
│ assistant: "再查 JB 数据..."                     │
│ tool result: [最多 6000 字符]  ← historyChars   │
│ ─────────── 当前轮 ───────────                   │
│ tool result: [最多 12000 字符] ← maxChars       │
└─────────────────────────────────────────────────┘
```

### 压缩触发后（历史 > 40 条）

```
┌─────────────────────────────────────────────────┐
│ system prompt（固定）                            │
│ 【历史对话摘要】早期对话的压缩文字...            │
│ ─────────── 最近 20 条完整消息 ────────────      │
│ user: "..."                                      │
│ tool result: [最多 6000 字符]                    │
│ assistant: "..."                                 │
│ ...（共 20 条）                                  │
│ ─────────── 当前轮 ───────────                   │
│ tool result: [最多 12000 字符] ← maxChars       │
└─────────────────────────────────────────────────┘
```

> **结论**：你说的"20条 + 压缩消息"只在触发压缩之后才成立。  
> 压缩前是全量历史（1~40条完整原文），压缩后才变为"摘要 + 最近20条"。

### 累积摘要机制

压缩不是一次性的，而是**滚动叠加**：

```
第一次压缩（历史超 40 条）：
  msg1~msg21 → summary_1

第二次压缩（继续聊，历史又超 40 条）：
  summary_1 + msg22~msg41 → summary_2（摘要叠入摘要）

以此类推，摘要始终只有一段文字，不会无限膨胀
```

## 通俗举例

**场景：** 你和 AI 对话，先问触发情况，再追问 BIN 分布，最后问某片的详情。

---

### 第 1 轮

> 用户：WA03P02G 最近触发了几次？

AI 调用工具，工具返回 15000 字符的 JSON 数据：

```
工具原始数据：15000 字符
  ↓ toolResultMaxChars = 12000
发给 LLM 分析：12000 字符（截断了尾部 3000）
  ↓ toolResultMaxHistoryChars = 6000
存入历史：6000 字符（进一步截断）

AI 回答："共触发 12 次，最近一次 5 月 20 日"
```

---

### 第 2 轮

> 用户：BIN7 有几颗？

AI 发给 LLM 的内容：

```
system prompt
+ 第1轮 user 消息
+ 第1轮 assistant 消息
+ 第1轮 tool result（6000 字符，历史截断版）← 不完整
+ 第1轮 assistant 结论
+ 当前轮 tool result（最多 12000 字符）← 新查的，完整版
```

AI 用新查的完整数据回答 BIN7 问题，第 1 轮的历史数据虽然不完整，但不影响这轮。

---

### 第 40+ 轮（触发压缩）

对话已经很长，历史超过 40 条消息：

```
压缩前：
  msg1  msg2  msg3 ... msg20  msg21 ... msg40  msg41

压缩后：
  [summary: "用户先后查询了 WA03P02G 触发情况（12次）、
             BIN7 分布（3颗）、lot ABCD1234 第5片详情..."]
  msg21 ... msg40  msg41   ← 最近 20 条保留原文
```

后续轮次 LLM 通过摘要了解早期对话背景，但细节（具体 lot 号、精确数字）可能已丢失。

---

## 调参建议

| 场景 | 建议 |
|------|------|
| 追问历史工具数据时 AI 回答不准 | 把 `toolResultMaxHistoryChars` 调到 12000（与当轮一致） |
| 长对话响应越来越慢 | 把 `toolResultMaxHistoryChars` 调低到 2000–3000 |
| 单次查询 AI 只看到部分数据 | 把 `toolResultMaxChars` 调高到 20000–30000 |
| 模型上下文 128k（如 MiniMax） | 两个值都可以调到 12000，token 完全撑得住 |
