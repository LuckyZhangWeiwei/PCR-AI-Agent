# PCR AI Agent 功能测试提问集

> 每条提问直接复制粘贴到 AI Agent 对话框即可触发对应工具。
>
> **数据说明：**
> - **Yield Monitor 数据** → 来自 `YMWEB_YIELDMONITORTRIGGER`（类型固定为 `delta_diff`，v3 自动过滤）
> - **JB STAR 数据** → 来自 `INFCONTROL ⋈ INFLAYERBINLIST`（`PASSTYPE='TEST'`，v3 自动过滤）
> - 未指定时间范围时，API 默认查最近 **1 年**数据
> - 下方提问中的 `WA03P02G`、`NF12615.1X` 等是示例值，请替换为实际生产数据中的设备号/lot 号

## 1. query_yield_triggers — 查询 Yield Monitor 触发记录

> **对应函数：** `runTool("query_yield_triggers", ...)` → 内部调用 `parseYieldMonitorTriggerV3Query()` + Oracle/Dummy 列表路径
>
> **通俗理解：**
> Yield Monitor 会对每片晶圆测试数据做实时监控，一旦某项指标超出管控线就会产生一条"触发记录"（trigger）——比如 delta_diff 触发表示这片晶圆的 bin 分布相对上片发生了异常漂移。工程师每天要审查这些触发记录，判断是偶发异常还是工艺系统性漂移。这个工具可以用自然语言灵活过滤：按设备、lot、wafer 号、机台、探针卡、pass、时间范围，查出满足条件的触发记录列表。
>
> **关键参数说明：**
> - `device`：产品代码，如 `WA03P02G`
> - `lotId`：lot 号，如 `NF12615.1X`
> - `wafer`：晶圆槽位标识
> - `hostname`：测试机台名称
> - `probeCard`：探针卡 ID
> - `probeCardType`：探针卡类型
> - `pass`：pass ID（奇数 1/3/5 为正测）
> - `timeFrom` / `timeTo`：时间范围（ISO 8601 格式）
> - `limit`：最多返回条数（默认 50，最大 200）

**1a. 查询特定 lot 的所有触发**

```
NF12615.1X 这个 lot 共触发了多少次 Yield Monitor？分别是哪几片 wafer？
```

**1b. 查询特定设备最近触发**

```
最近 7 天 WA03P02G 这个器件的 Yield Monitor 触发了多少次？触发的 lot 号和 wafer 号是什么？
```

**1c. 按机台过滤**

```
帮我查一下最近 1 个月机台 TESTER01 上 WA03P02G 器件的 delta_diff 触发记录，有哪些 lot 触发了？
```

**1d. 查询特定 pass 的触发**

```
NF12615.1X lot 的 pass 1 正测触发了几条 Yield Monitor 记录？触发时间分别是什么？
```

**1e. 时间范围精确查询**

```
查询 2026-04-01 到 2026-04-30 期间所有器件的 Yield Monitor 触发记录，最多显示 100 条，按时间倒序。
```

## 2. aggregate_yield_triggers — 聚合 Yield Monitor 触发统计

> **对应函数：** `runTool("aggregate_yield_triggers", ...)` → 内部调用 v3 聚合路径（Node 层全量行集计算）
>
> **通俗理解：**
> 只看触发记录列表很难看出规律，聚合统计才能快速回答"哪个设备最容易触发？哪台机台问题最多？"这类问题。这个工具对触发记录按指定维度分组计数，支持按设备、机台、lot、wafer、探针卡、pass、日期等任意组合分组，并取 top N 排行。一次查询就能看清全局趋势——比如某台机台上周触发了 80% 的告警、某种探针卡的触发率是其他的 3 倍，这类结论在列表里几乎找不出来。
>
> **关键参数说明：**
> - `dimensions`：分组维度（逗号分隔），可选：`device`、`hostname`、`lotId`、`wafer`、`probeCard`、`pass`、`timeDay`
> - `groupTop`：只返回前 N 组（默认 10）
> - 其余过滤参数与 `query_yield_triggers` 相同

**2a. 按设备统计触发次数排行**

```
最近 1 个月哪些器件的 Yield Monitor 触发次数最多？按 device 维度排行，前 10 名。
```

**2b. 按机台统计触发次数**

```
过去 3 个月各测试机台的触发次数是多少？哪台机台告警最频繁？用柱状图展示。
```

**2c. 按 lot + device 联合分组**

```
WA03P02G 器件最近 30 天各 lot 分别触发了多少次 Yield Monitor？触发最多的是哪个 lot？
```

**2d. 按天统计触发趋势**

```
最近 14 天 Yield Monitor 每天的触发次数是多少？有没有哪天出现明显的异常高峰？生成折线图。
```

**2e. 按探针卡统计**

```
帮我统计所有探针卡的触发次数，哪块探针卡触发最频繁？是否与特定设备相关？
```

## 3. query_jb_bins — 查询 JB STAR 测试 Bin 记录

> **对应函数：** `runTool("query_jb_bins", ...)` → 内部调用 `parseInfcontrolLayerBinsV3Query()` + Oracle/Dummy 列表路径
>
> **通俗理解：**
> JB STAR 记录了每片晶圆在测试机台上完成 CP 测试后的 bin 分布——每个 bin 对应一种测试项目的通过/失败分类，BIN0/BIN1 通常为良品，BIN2 以上为各类失效。工程师拿到一份测试数据，首先要看"这片晶圆的 bin 分布是什么样的，良率多少，哪个 bin 失效最多"，这个工具完成的就是这个查询：给出每条测试记录的完整 bin 计数，支持按设备、lot、槽位、机台、探针卡、时间范围过滤。
>
> **关键参数说明：**
> - `device`：产品代码
> - `lot`：lot 号，如 `NF12615.1X`
> - `slot`：晶圆槽位（1–25）
> - `cardId`：探针卡 ID
> - `probeCardType`：探针卡类型
> - `testerId`：测试机台 ID
> - `passId`：pass ID
> - `meslot`：MES 槽位 ID
> - `testEndFrom` / `testEndTo`：测试结束时间范围
> - `limit`：最多返回条数（默认 50，最大 200）

**3a. 查询特定 lot 的测试记录**

```
NF12615.1X 这个 lot 的 JB STAR 测试记录有哪些？每片晶圆测试完的良率是多少？
```

**3b. 查询特定槽位的 bin 分布**

```
NF12615.1X lot 槽位 5 这片晶圆 JB STAR 测试结果的 bin 分布是什么？各 bin 各有多少颗 die？
```

**3c. 按机台查询最近测试**

```
机台 TESTER02 最近 3 天测试了哪些晶圆？测试结果中良率最低的是哪片？
```

**3d. 查询特定设备最近测试**

```
WA03P02G 器件上周的 JB STAR 测试记录有哪些？哪些 lot 参与了测试？
```

**3e. 查询特定时间范围**

```
查 2026-05-01 到 2026-05-15 期间所有晶圆的 JB STAR 测试记录，按测试结束时间倒序显示最近 50 条。
```

## 4. aggregate_jb_bins — 聚合 JB STAR Bin 统计

> **对应函数：** `runTool("aggregate_jb_bins", ...)` → 内部调用 v3 聚合路径（`groupBy` 必须包含 `bin`）
>
> **通俗理解：**
> 看完 JB STAR 的列表记录，还需要知道"从更大范围来看，哪个 bin 失效最严重？哪台机台的 BIN2 失效最多？"。聚合工具把多片晶圆的 bin 计数按指定维度汇总，直接回答这类问题。尤其重要的是：它自动识别 PASSBIN 中定义的良品 bin 编号，只统计坏 bin 的计数，避免把良品数量混入失效分析。工程师用这个工具可以快速找出"主力失效 bin"和"哪些生产因素与该 bin 相关"。
>
> **关键参数说明：**
> - `groupBy`：分组维度（逗号分隔，`bin` 必须包含，自动补充），可选：`bin`、`device`、`lot`、`testerId`、`cardId`、`passId`、`slot`、`layerName` 等
> - `groupTop`：只返回前 N 组（默认 10）
> - 其余过滤参数与 `query_jb_bins` 相同

**4a. 统计各 bin 失效计数排行**

```
WA03P02G 器件最近 1 个月哪个失效 bin 的 die 数最多？按 bin 维度排行，生成柱状图。
```

**4b. 按机台 + bin 联合分析**

```
过去 3 个月各测试机台上 WA03P02G 器件的 BIN2 失效数量是多少？哪台机台 BIN2 最多？
```

**4c. 按 lot 统计失效 bin 分布**

```
NF12615.1X lot 的 JB STAR 测试结果中，各 bin 的失效计数是多少？哪个 bin 是主力失效？
```

**4d. 按探针卡 + bin 统计（探针卡质量分析）**

```
过去 2 个月哪块探针卡对应的 BIN3 失效最多？是否存在某块探针卡失效率明显偏高？
```

**4e. 失效 bin 趋势分析（按天）**

```
WA03P02G 器件近 30 天 BIN2 失效数量的每日趋势是什么？有没有哪天出现异常高峰？生成折线图。
```

## 5. generate_chart — 生成 ECharts 图表

> **对应函数：** `runTool("generate_chart", ...)` → 纯计算，不访问数据库；返回 ECharts option JSON，前端直接渲染
>
> **通俗理解：**
> 光看数字表格很难发现趋势和异常，图表让数据一目了然。这个工具把 AI 查询到的数据转换为标准的柱状图、折线图、饼图或散点图，并直接嵌入在对话气泡里渲染显示——不需要下载文件、不需要打开 Excel，在 AI Agent 界面里就能看到可视化结果。一般由 AI 主动判断在查询到数据后调用此工具，用户也可以明确要求"生成图表"或"用柱状图展示"。
>
> **关键参数说明：**
> - `chartType`：图表类型，`"bar"`（柱状图）、`"line"`（折线图）、`"pie"`（饼图）、`"scatter"`（散点图）
> - `title`：图表标题
> - `data.labels`：X 轴/分组标签数组
> - `data.series`：数据系列数组，每个系列有 `name` 和 `values` 数组
>
> **注：** generate_chart 通常不需要单独调用——在查询到数据后，AI 会自动判断是否生成图表。下面的提问示范的是"明确要求图表"时的语法。

**5a. 已有数据后明确要求图表**

```
把上面查到的各设备触发次数结果用饼图展示出来，每个设备占比多少。
```

**5b. 要求查询同时生成图表**

```
统计 WA03P02G 器件过去 7 天每天的 Yield Monitor 触发次数，然后用折线图展示趋势。
```

**5c. 多系列图表**

```
比较最近 1 个月 BIN2、BIN3、BIN4 各自的失效总数，用柱状图展示每个 bin 的数量，让我看哪个 bin 问题最严重。
```

**5d. 散点图查相关性**

```
统计过去 2 个月各 lot 的 Yield Monitor 触发次数和 JB STAR BIN2 失效数，用散点图展示两者是否有相关性。
```

## 6. ask_clarification — 问询澄清

> **对应函数：** `runTool("ask_clarification", ...)` → 返回澄清问题给用户，不访问数据库；AI 主动调用，用户无法直接触发
>
> **通俗理解：**
> 当用户的问题表达不明确、有歧义、或者缺少必要的过滤条件时（比如说"帮我查数据"而没有告诉 AI 查什么设备或什么时间），AI 会主动用这个工具提出一个具体问题，要求用户补充信息，而不是猜测后查出大量无关数据，或给出误导性的结论。这是保证分析准确的安全阀。
>
> **触发场景示例：**
> - 同一设备有多种 pass，用户没有指定查哪个 pass 但不同 pass 结果差异很大
> - 时间范围未指定，但数据量极大（全量查询可能超限）
> - 用户说"最近"但没有具体时间，AI 需要确认是"最近 7 天"还是"最近 1 个月"
> - 提到的 lot 号 AI 无法区分 Yield Monitor 还是 JB STAR 的查询意图
>
> **触发此工具的不明确提问示例（以下提问会让 AI 主动反问）：**

**6a. 缺少设备信息（过于宽泛）**

```
帮我查一下触发记录。
```

**6b. 缺少时间范围（数据量不确定）**

```
把所有 JB STAR 数据给我查出来。
```

**6c. 意图不明（Yield Monitor 还是 JB STAR？）**

```
NF12615.1X 这个 lot 的测试结果怎么样？
```

**6d. 多义维度（需要确认分析角度）**

```
WA03P02G 这个设备的问题在哪？
```

## 综合场景提问（多轮对话 + 多工具调用）

以下提问会触发 AI 连续调用多个工具，模拟真实分析流程。

**场景 A：设备良率问题排查流程**

```
WA03P02G 器件最近 1 个月的 Yield Monitor 触发情况如何？触发最多的是哪个 lot？这个 lot 的 JB STAR BIN 分布又是什么样的？帮我全面分析一下这个器件近期是否有良率异常。
```

**场景 B：机台对比分析**

```
对比 TESTER01 和 TESTER02 这两台机台最近 2 个月的 Yield Monitor 触发次数和 JB STAR BIN2 失效数，哪台机台表现更差？用柱状图对比展示，并给出你的判断。
```

**场景 C：探针卡质量评估**

```
帮我评估过去 3 个月各探针卡的质量：分别看它们对应的 Yield Monitor 触发次数和 JB STAR BIN2/BIN3 失效数，哪块探针卡问题最多？需要重点关注哪块？
```

**场景 D：时序趋势分析**

```
过去 30 天 Yield Monitor 每天的触发次数是多少？有没有某几天出现集中爆发？生成折线图，并告诉我异常峰值出现在哪一天、对应哪些 lot。
```

**场景 E：快速问题定位**

```
最近一周有没有什么比较异常的测试情况？帮我快速梳理一下 Yield Monitor 和 JB STAR 两边的数据，看看有没有需要立即关注的问题。
```
